# 📘 [AI 헬스케어] 웹 기초(HTML) & GitHub

## Part 1. HTML 핵심 요약

HTML은 웹 페이지의 **'뼈대'**를 만드는 마크업 언어입니다. 
서버가 보내준 설계도(HTML)를 브라우저가 읽어서 화면을 그려줍니다.

### 1. HTML 기본 구조 (The Skeleton)
모든 HTML 문서는 이 구조를 따릅니다.
- `<!DOCTYPE html>`: 이 문서는 HTML5 버전임을 선언.
- `<html>`: 문서의 시작과 끝.
- `<head>`: 브라우저에게 주는 정보 (제목, 설정 등 / 화면엔 안 나옴).
- `<body>`: **실제 화면에 보이는 내용**이 들어가는 곳.

### 2. 필수 태그 정리
- **제목:** `<h1>` (가장 큼) ~ `<h6>` (가장 작음)
- **문단:** `<p>` (Paragraph, 줄 바꿈이 포함됨)
- **목록:** - `<ul>`: 순서 없는 목록 (점)
  - `<ol>`: 순서 있는 목록 (1, 2, 3...)
  - `<li>`: 목록의 각 항목
- **링크:** `<a href="https://google.com">구글로 이동</a>`
- **이미지:** `<img src="이미지주소.jpg" alt="이미지설명">` (닫는 태그 없음)

### 3. Form 태그 (데이터 전송의 핵심)
사용자의 입력을 받아 서버로 보내는 **'택배 상자'** 역할입니다.

#### ① Form 태그의 속성
```html
<form action="서버주소" method="전송방식">
```
- **action:** 데이터를 받을 목적지 URL (예: `/login`, `/search`).
- **method:** - `GET`: 데이터를 주소창에 노출해서 전송 (검색, 조회용). 빠르고 캐싱 가능.
  - `POST`: 데이터를 숨겨서 전송 (로그인, 회원가입, 글쓰기). 보안 유지.

#### ② Input과 Label (입력창)
```html
<label for="username">아이디</label>
<input type="text" id="username" name="user_id">
```
- **`<label>`:** 시각장애인 스크린 리더와 클릭 편의성을 위해 사용. `for` 속성과 `input`의 `id`를 일치시켜야 함.
- **`<input>`:** 실제 입력받는 상자. `type`에 따라 모양이 변함 (`text`, `password`, `checkbox`, `radio` 등).
- **`name` 속성 (중요!):** 서버가 데이터를 식별하는 이름표. 이게 없으면 서버로 데이터가 안 날아감.

## Part 2. GitHub 협업의 정석 (A to Z)

단순히 `push`만 하는 것이 아니라, **'이슈(Issue)'를 만들고 'PR(Pull Request)'로 검사받는** 실제 현업의 프로세스입니다.

### 📋 1단계: 할 일 등록 (GitHub Issue)
코드를 짜기 전에 "내가 뭘 할 건지" 팀원들에게 알리는 단계입니다.

1. GitHub 저장소 상단 메뉴의 **[Issues]** 탭 클릭.
2. **[New issue]** 버튼 클릭.
3. 제목 예시: `feat: 환자 검색 기능 구현`
4. 내용 작성 후 **Submit**.
5. **이슈 번호(#1, #2 등)를 기억하세요!**

### 🌿 2단계: 작업 공간 분리 (Local Branch)
`main` 브랜치는 건드리지 않고, 나만의 작업 공간을 만듭니다.

```bash
# 1. 터미널 열기 (프로젝트 폴더에서)
# 2. 최신 상태 동기화 (습관적으로 하기)
git checkout main
git pull origin main

# 3. 브랜치 생성 및 이동
# 보통 '기능명'이나 '이슈번호'를 포함해서 짓습니다.
git checkout -b feature/search-patient
```

### 🔨 3단계: 코딩 및 기록 (Commit)
내 컴퓨터에서 작업을 진행합니다.

```bash
# ...HTML/Python 코드 작성...

# 1. 상태 확인
git status

# 2. 장바구니 담기
git add .

# 3. 기록 남기기 (메시지 중요!)
# 팁: 이슈 번호를 남기면 나중에 추적하기 좋습니다.
git commit -m "feat: 검색 form 태그 추가 (#1)"
```

### 🚀 4단계: 깃허브로 전송 (Push)
내 작업 공간(브랜치)을 원격 저장소로 보냅니다.

```bash
# git push origin [내브랜치이름]
git push origin feature/search-patient
```

### 💌 5단계: 검사 요청 (Pull Request)
"제 코드가 완성되었으니 `main`에 합쳐주세요!"라고 요청하는 단계입니다.

1. GitHub 저장소 메인 페이지 접속.
2. **"Compare & pull request"** 초록색 버튼이 자동으로 뜹니다. (안 뜨면 [Pull requests] 탭 -> New PR).
3. **내용 작성 꿀팁:**
   - 내용에 `Closes #1`이라고 적으면, 이 PR이 머지될 때 1번 이슈가 자동으로 닫힙니다 (자동 완료 처리).
4. **[Create pull request]** 클릭.
5. 팀원들이 코드를 리뷰하고 댓글을 답니다.

### 🤝 6단계: 병합 (Merge)
팀장의 승인이 떨어지면 병합합니다.

1. PR 페이지 하단의 **[Merge pull request]** 버튼 클릭 -> **[Confirm merge]**.
2. 이제 내 코드가 `main` 브랜치에 정식으로 반영되었습니다.
3. (선택) GitHub 상에서 다 쓴 브랜치(`feature/search-patient`)는 삭제(Delete branch)해도 됩니다.

### 🔄 7단계: 내 컴퓨터 동기화 (Local Update)
GitHub의 `main`은 최신 버전이 되었지만, 내 컴퓨터의 `main`은 아직 옛날 버전입니다.

```bash
# 1. 메인 브랜치로 복귀
git checkout main

# 2. 최신 내용 당겨오기 (가장 중요!)
git pull origin main

# 3. 다 쓴 로컬 브랜치 삭제 (정리용)
git branch -d feature/search-patient
